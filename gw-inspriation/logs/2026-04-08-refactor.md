**设计要点：**
- Content 管线：Goldmark（MD→HTML）→ Bluemonday（`UGCPolicy` XSS 净化）→ 存库
- `ContentRaw` 字段保留原始 Markdown，供编辑时回显；渲染 HTML 存 `Content` 字段
- 热度公式（Hacker News 变体）：`HotScore(replies, votes int, createdAt) float64` = `(replies*2 + votes) / pow(ageHours+2, 1.5)` — 已导出，方便单元测试
- 楼层序号：事务内 `SELECT posts FOR UPDATE` + `Number = RepliesCount + 1`（悲观锁，并发安全）
- 帖子 Post 反规范化：`RepliesCount`、`LastReplyAt`、`LastReplyUser`（来自 Flarum 设计）
- `search_vector tsvector` 列带 `gorm:"-:migration"` 跳过 AutoMigrate；`MigrateSQL()` 返回 PostgreSQL 触发器 DDL，**需部署后手动执行一次**
- MVP 搜索降级：全文搜索不可用时退回 `ILIKE`
- `sensitive` 过滤（importcjj/sensitive）留作后续 TODO，MVP 未引入

**参考来源：** Flarum 的 RepliesCount/LastReplyAt 设计 + Gitea goldmark/bluemonday 管线

**验证：** `go build ./...` 通过。

**API 端点：**
```
POST   /api/social/reactions/:target_type/:target_id/:type
DELETE /api/social/reactions/:target_type/:target_id/:type
GET    /api/social/reactions/counts?targets=...
GET    /api/social/reactions/mine/:target_type/:target_id
```

**设计要点：**
- UNIQUE INDEX 通过原生 SQL 建立（GORM 不支持多列唯一索引 tag）
- `syncCount` 在 like 操作后原子更新目标表 `vote_up`（`GREATEST(0, vote_up ± 1)`）
- `syncCount` 静默失败，不影响主流程（comment/forum 表尚未创建时不报错）
- 匿名用户只读，写操作返回 401
- `favorite` 不影响 `vote_up`（收藏逻辑与热度排序分离）

**参考来源：** Gitea reaction.go 的 UNIQUE(s) 约束 + Artalk vote.go 的 syncCount 模式

**验证：** `go build ./...` 通过。
