## 一、哪些批评是准确的，直接接受

### 1.1 安全阻断项（2/10）— 完全接受

评委说的是实话。`LLMProfile.api_key` 明文存储、`X-Account-ID` 无签名这两个问题不是"技术债务"，是**上线前必须关闭的硬性阻断项**。

当前阶段（内测）可以接受，因为：
- 数据库只在本地/内网，没有公网暴露
- `X-Account-ID` 伪造在内测环境里没有实际攻击面

但这不改变结论：**P-4A 和 P-4B 是 Phase 4 的第一优先级，不能被其他 Phase 4 工作并行推进时跳过**。

### 1.2 Background Job 不持久化（5/10 运行时可靠性）— 完全接受

评委的措辞是准确的："不是低概率问题，是任何生产部署都会遇到的确定性问题"。Memory Worker 的 goroutine 在进程重启时丢失任务，这在内测阶段影响有限（记忆整合失败只是少一次摘要），但在正式运营时会造成用户可感知的体验退化。

P-4G（Background Job Runtime，DB 持久化）的优先级需要提升，见下文计划调整。

### 1.3 `game_loop.go` 职责过重 — 接受，但不急

评委说"这是现在的问题，不是未来风险"。这个判断是对的，但影响范围是**测试覆盖困难**，不是功能错误。当前阶段没有单元测试覆盖 `game_loop.go` 的 LLM 调用路径，这是已知的技术债务，Phase 4 拆分时处理。

## 二、哪些批评需要补充背景

### 2.1 Token 计数精度（5/10）

评委说 ±15% 误差对 Token Budget 操作是问题。这个批评在**长上下文场景**下成立，但对当前 WE 的实际使用场景需要补充：

- 当前 `LLM_TOKEN_BUDGET=2000`，实际上下文窗口 128K（GLM-4-Plus）
- Token Budget 的作用是**防止世界书词条撑爆上下文**，不是精确控制到 token 级别
- ±15% 的误差在 2000 token 预算下是 ±300 token，对于"是否触发裁剪"的判断影响可接受

**结论**：Token 精度问题在当前阶段不是阻断项。当 WE 需要支持 4K 以下上下文窗口的模型（如某些本地模型）时，精度问题才会变成实际 bug。届时引入 provider-specific 分词器（tiktoken for OpenAI，sentencepiece for Gemini）。

### 2.2 并发保护只有 reject 模式

评委对比了 TH 的乐观锁 + CAS。WE 的 `SELECT ... FOR UPDATE` + `generating bool` 方案在**单实例部署**下是正确的，不是"粗暴"——它保证了原子性。

`queue` 模式（排队等待而非拒绝）是用户体验优化，不是正确性问题。当前 `reject` 模式下，前端收到 409 后可以提示用户"正在生成中，请稍候"，这是可接受的 UX。

**结论**：queue 模式是 Phase 4 的优化项，不是阻断项。

### 2.3 MCP 方向不清晰（3/10）

评委的批评准确：触发条件模糊，没有具体接口设计。

但 WE 的定位是**游戏发布平台引擎**，不是通用 AI 助手框架。MCP 的核心价值（接入本地文件系统、代码执行）对游戏创作场景的价值有限——创作者更需要的是"AI 帮我修改世界书词条"（已有 `resource:*` 工具），而不是"AI 帮我读本地文件"。

**结论**：MCP 继续暂缓，但需要在 P-WE-OVERVIEW.md 中明确写出"不做的理由"，而不是"触发条件模糊"。见下文计划调整。

### 2.4 RAG 方向（6/10）

评委说"没有进入正式里程碑"。这是有意为之的：RAG 的价值在常驻角色（>500 回合）场景，而常驻角色功能本身还没有进入路线图。在没有用户验证的情况下提前实现 RAG 是过度工程。

**结论**：RAG 的技术方案（pgvector + `GetForInjection` mode 参数）已经设计完整，等常驻角色进入路线图时一并推进。

## 三、评委提供的扩展设计思路评估

### 3.1 MCP 最小路径

评委建议：在现有 `Tool` 接口上增加 `MCPToolProvider`，实现 stdio + Streamable HTTP 传输，复用 `Registry.RegisterProvider()`，安全上参考 TH 的 Deferred 执行模型（`require_user_confirm` 等级）。

这个设计思路是合理的，但**当前不做**。原因：
1. Preset Tool（HTTP 回调）已经覆盖了云端集成场景
2. 本地工具（文件系统、代码执行）在游戏创作场景的需求尚未被用户验证
3. Deferred 执行模型需要前端配合（用户确认弹窗），GW 前端目前不具备这个能力

记录在案，等 GW 前端成熟后重新评估。

### 3.2 RAG 接入点

评委建议：在 `GetForInjection()` 增加 `mode` 参数，`mode=semantic` 时调用 embedding API + pgvector 检索。这与之前的设计思路完全一致，接受。

## 四、计划文档调整决策

### 4.1 P-WE-OVERVIEW.md 需要调整的内容

**调整 1：Phase 4 优先级排序明确化**

当前 Phase 4 的条目没有内部优先级排序，评委的批评揭示了一个问题：P-4A（API Key 加密）和 P-4B（JWT）是**其他所有 Phase 4 工作的前置条件**，必须先完成。

新的 Phase 4 执行顺序：
```
第一批（上线阻断，必须先完成）：
  P-4A  API Key AES-256-GCM 加密
  P-4B  JWT Auth

第二批（平台工程，可并行）：
  P-4C  多 Provider 原生适配
  P-4D  OpenAPI 文档
  P-4E  对话导入/导出
  P-4G  Background Job Runtime（优先级提升，见下）

第三批（体验增强）：
  P-4F  双层记忆压缩
  P-4H  Floor Run Phase SSE
  P-4I  VN 渲染资产系统
```

**调整 2：P-4G 优先级提升**

Background Job 不持久化是确定性的生产问题，不是低概率风险。P-4G 从"第三批"提升到"第二批"，与 P-4C/P-4D/P-4E 并行推进。

**调整 3：P-3H（MCP）暂缓理由明确化**

当前写的是"触发条件：创作者需要接入本地 MCP 工具时再做"，这太模糊。改为明确的不做理由：
- WE 定位是游戏发布平台引擎，本地工具接入不在核心场景
- Preset Tool（HTTP 回调）已覆盖云端集成
- 等 GW 前端具备用户确认交互能力后重新评估

**调整 4：Token 精度问题的触发条件明确化**

在 P-WE-OVERVIEW.md 的技术债务表中，Token 计数精度问题补充触发条件：当需要支持 4K 以下上下文窗口的模型时，引入 provider-specific 分词器。

## 五、不调整的内容及理由

**不提前做 P-4A/P-4B**：当前处于内测阶段，数据库不公网暴露，安全风险可控。但 Phase 4 开始时这两项必须第一批完成。

**不提前做 queue 模式**：reject 模式在单实例部署下功能正确，queue 模式是 UX 优化，Phase 4 处理。

**不提前做 RAG**：常驻角色场景未验证，过度工程。

**不做 MCP**：游戏平台场景需求未验证，Preset Tool 已覆盖主要集成场景。

## 六、对评分的整体判断

评委的评分框架是"面向市场的完整商品底座"，这个框架下 2/10 的安全评分是准确的。

但 WE 当前的实际阶段是**内测引擎**，不是"面向市场的完整商品底座"。在这个阶段，正确的优先级是：

1. 让内测用户能玩到游戏（端到端流程跑通）
2. 收集内测反馈，验证游戏包格式和 GW 前端设计
3. 在进入正式运营前完成 P-4A/P-4B

评委的批评是对"Phase 4 完成后的 WE"的预期，而不是对"当前 WE"的否定。差距是真实的，路线图是清晰的，执行顺序需要按上文调整。
